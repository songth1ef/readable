# Readable / 可读地 需求文档

## 1. 产品定位

**Readable**，中文名暂定 **可读地**。

一句话定位：把一段很长、很散的文案，生成一个结构清楚、排版好看、可以访问和分享的 HTML 阅读页。

它不是公众号编辑器，也不是普通 Markdown 编辑器。它更像一个“长文阅读页生成器”：用户只管输入内容，系统负责润色、拆结构、套模板、生成页面。

## 2. 要解决的问题

很多长文从上到下读起来很累，尤其像微信公众号长文那种形式：

- 内容很长，但缺少导航。
- 层级不清楚，读者很难快速扫读。
- 重点、步骤、代码、列表混在一起。
- 分享后只能按原始长文阅读，体验比较单一。
- 用户有文案，但不想自己设计页面或写 HTML。

Readable 要做的是：把原始文案变成一个更适合阅读、复盘、分享的网页。

## 3. 核心流程

1. 用户打开网站。
2. 用户粘贴一段原始文案。
3. 用户选择一个 HTML 样式模板。
4. 用户点击生成。
5. 系统调用一个便宜、快速的 LLM，把文案润色并整理成结构化 JSON。
6. 前端模板读取 JSON，渲染成一个完整的 HTML 阅读页。
7. 系统为生成结果分配一个 ID。
8. 用户可以通过这个 ID 对应的链接访问页面。
9. 未登录用户可以免费生成一次并预览结果。
10. 如果要查看稳定页面、保存、下载或后续管理，需要登录。

## 4. MVP 必须做的功能

第一版只做最小可用产品，目标是尽快跑通“输入文案 -> 生成阅读页 -> 登录保存”的闭环。

必须包含：

- 文案输入框。
- 模板选择入口。
- 至少一个固定 HTML 阅读模板。
- 调用 LLM，把文案转成结构化 JSON。
- 用模板把 JSON 渲染成阅读页。
- 每个生成结果都有唯一 ID。
- 通过 `/r/{id}` 访问生成页面。
- 未登录用户可以生成一次。
- 查看稳定结果、保存结果、下载 HTML 时要求登录。
- 支持第三方登录，优先 Google 登录。
- 登录后可以看到自己的生成记录。

## 5. 暂时不做的功能

这些功能有价值，但不放进第一版：

- 用户自定义模板。
- AI 自动生成模板样式。
- 模板市场。
- 团队协作。
- 自定义域名。
- PDF 导出。
- 复杂的富文本编辑器。
- 很细的计费系统。

## 6. 登录方案

第一版确定使用 **Clerk**。

原因：

- 接入速度快。
- 对 Next.js 支持很好。
- 可以快速开启 Google 登录。
- 用户管理后台现成。
- 很适合 MVP 阶段，不需要自己写一套登录系统。
- 比较适合先把“登录体验”快速做好。

备选方案只保留在后续参考中，不作为当前实现方向：

- **Auth.js**：更自由，但配置和维护成本更高。
- **Supabase Auth**：如果未来希望认证和数据库更一体化，可以再评估。
- **Firebase Auth**：成熟稳定，但整体产品栈会更偏 Firebase。

当前结论：账号系统优先使用 `Clerk`，不要在第一版花太多时间自建认证。

## 7. 未登录免费生成一次

产品策略：不要一开始就强迫登录。用户应该先体验到“生成结果确实好看、有用”，再提示登录。

规则：

- 未登录用户可以免费生成一次。
- 生成后可以看到一次预览。
- 如果要打开正式结果页、保存、下载 HTML 或再次生成，就需要登录。
- 登录后，可以把刚才未登录生成的结果绑定到当前账号。

实现思路：

- 浏览器里保存一个匿名试用标记，比如 cookie 或 localStorage。
- 服务端也记录一个匿名 token 的 hash。
- 可以额外记录 IP hash 和 user-agent hash，用来降低滥用。
- 未登录生成的文档先作为临时文档保存。
- 用户登录后，把临时文档转移到这个用户名下。

## 8. LLM 选择

这个项目不需要特别聪明的大模型。它主要做三件事：

- 轻度润色。
- 拆分结构。
- 按指定 JSON 格式输出。

所以模型选择优先级应该是：

1. 便宜。
2. 快。
3. JSON 输出稳定。
4. 中文表现够用。
5. 接口稳定。

建议第一版只封装一个模型，不要一开始做复杂的多模型切换。

可以考虑：

- OpenAI 的小模型或 mini 模型。
- DeepSeek 等低成本模型。
- OpenRouter，方便后续切换模型。

当前建议：代码里做一个 `llmProvider` 适配层，第一版只接一个便宜快速的模型。以后如果价格、速度或效果不满意，再替换底层模型。

## 9. AI 输出格式

LLM **不要直接生成 HTML**。

LLM 只负责输出结构化 JSON。模板负责页面样式、交互、导航、复制按钮、响应式布局等。

这样做的好处：

- 页面风格稳定。
- 不容易生成乱七八糟的 HTML。
- 模板可以复用。
- 后续可以换模板，不需要重新理解原始文案。
- 安全性更好。

初版 JSON 结构可以先设计成这样：

```json
{
  "meta": {
    "title": "文档标题",
    "subtitle": "一句话摘要",
    "tag": "指南",
    "language": "zh-CN"
  },
  "sections": [
    {
      "id": "section-1",
      "title": "章节标题",
      "summary": "这一节的简短说明",
      "blocks": [
        {
          "type": "paragraph",
          "content": "润色后的段落内容"
        },
        {
          "type": "list",
          "items": ["要点一", "要点二"]
        },
        {
          "type": "code",
          "language": "bash",
          "content": "npm run build"
        }
      ]
    }
  ]
}
```

服务端需要校验这个 JSON：

- 字段是否完整。
- 类型是否正确。
- section 数量是否合理。
- block 类型是否在允许范围内。
- 文本长度是否超限。

如果模型输出不合法，可以自动修复一次。修复失败就返回生成失败状态。

## 10. 模板系统

第一版模板系统尽量简单：

- 先做固定模板。
- 模板代码放在项目里。
- 模板只接收结构化 JSON。
- 模板不执行用户输入的脚本。
- 模板负责渲染最终阅读页。

模板应包含这些元信息：

- 模板 ID。
- 模板名称。
- 模板描述。
- 模板版本。
- 预览图。
- 支持的 JSON schema 版本。

现有的 `html-doc` skill 可以作为第一个模板的参考：

- 固定 HTML 骨架。
- 固定 CSS 和运行时逻辑。
- 每次只替换文档标题和结构化内容。
- 不让 AI 每次重新写页面样式。

后续如果要做“AI 生成模板”，建议不要让 AI 直接生成任意 HTML/JS，而是生成受限制的模板配置或模板 DSL。

## 11. 文档状态

每个生成结果可以有这些状态：

- `draft`：已生成，但还没有正式发布。
- `published`：已经有稳定访问链接。
- `archived`：用户归档，不在默认列表里显示。
- `failed`：生成失败。

ID 设计：

- 数据库内部用 UUID 或 CUID。
- 对外访问用短 ID。
- 访问路径示例：`/r/{publicId}`。

比如：

```text
/r/abc123
```

## 12. 页面访问规则

第一版建议：

- 未登录用户可以生成一次。
- 未登录用户可以看生成后的临时预览。
- 打开稳定链接 `/r/{id}` 需要登录。
- 下载 HTML 需要登录。
- 查看个人文档列表需要登录。

后续可以扩展：

- 文档默认私有。
- 用户可以手动设置公开分享。
- 公开文档允许未登录访问。
- 私有文档只有作者能访问。

## 13. 技术栈建议

当前确定的第一版技术方向：

- 前端和后端：Next.js。
- 样式：Tailwind CSS。
- 登录：Clerk。
- 数据库：Postgres。
- ORM：Prisma 或 Drizzle。
- 部署：腾讯云。
- CDN 与全球访问加速：Cloudflare。
- 支付：先接国际支付。
- LLM：先封装一个便宜快速的模型。

为什么这样选：

- Next.js 适合快速做 SaaS 型产品。
- Clerk 能最快把 Google 等第三方登录接起来。
- Postgres 足够稳定，也不会把业务过度绑定到某个平台。
- 腾讯云部署更符合后续如果要兼顾国内访问的思路。
- Cloudflare 可以先承担 CDN、缓存、基础安全和海外访问加速。
- 第一版先走国际支付，把主流程跑通。

当前部署策略：

- 先采用“腾讯云部署 + Cloudflare CDN”的组合。
- 目标是先让国内外用户都尽量能访问。
- 第一阶段先不因为备案问题阻塞上线。
- 如果后续用户量上来，且大陆访问成为核心需求，再正式评估备案和更完整的中国区基础设施方案。

## 15. 关键页面

### 生成页

用户主要工作的地方。

包含：

- 文案输入区。
- 模板选择区。
- 生成按钮。
- 登录入口。
- 免费试用提示。

### 生成结果页

生成完成后展示。

包含：

- 阅读页预览。
- 登录提示。
- 重新生成按钮。
- 下载 HTML 按钮。
- 复制链接按钮。

### 阅读页

路径：`/r/{publicId}`。

包含：

- 文档标题。
- 摘要。
- 章节导航。
- 正文内容。
- 代码块。
- 阅读进度。
- 移动端适配。

### 个人中心

登录后查看自己的生成历史。

包含：

- 文档列表。
- 文档状态。
- 创建时间。
- 打开页面。
- 复制链接。
- 归档。

## 16. 支付方案

第一版支付策略：

- 先只做国际支付。
- 不在第一版同时接入微信支付、支付宝等国内支付方式。
- 优先目标是尽快验证产品是否有人愿意付费。

建议方向：

- 使用 Stripe Checkout 做支付页。
- 如果后续需要订阅制，再接 Stripe Billing。
- 支付完成后，通过 webhook 更新用户套餐状态或额度状态。

为什么先这样做：

- 接入快。
- SaaS 产品验证成本低。
- 不需要一开始就处理国内支付链路带来的大量额外复杂度。

后续再考虑：

- 微信支付。
- 支付宝。
- 国内主体、结算、发票、税务和相关合规问题。

## 17. 开放问题

这些问题后续需要继续确认：

- 登录后生成的页面，默认公开还是默认私有？
- 未登录生成后，是展示完整预览，还是只展示部分预览？
- 原始文案是否长期保存？
- 国内网络环境下，优先选哪个 LLM 服务商？
- 第一版是否一定要支持 HTML 文件下载？
- 模板是先写死在代码里，还是一开始就入库？
- 是否需要支持 Markdown 输入？
- 是否需要支持代码块语言自动识别？
- 在未备案前，腾讯云源站 + Cloudflare CDN 的实际大陆访问效果是否满足第一版要求？

## 18. MVP 里程碑

### 阶段一：项目骨架

- 初始化 Next.js 项目。
- 接入 Clerk 登录。
- 建立数据库。
- 做出生成页。

### 阶段二：AI 生成

- 设计 prompt。
- 设计 JSON schema。
- 接入 LLM。
- 校验和修复 JSON。
- 保存生成结果。

### 阶段三：阅读模板

- 做第一个固定模板。
- 根据 JSON 渲染页面。
- 支持 `/r/{id}` 访问。
- 支持下载 HTML。

### 阶段四：用户体验

- 未登录免费生成一次。
- 登录后认领生成结果。
- 个人文档列表。
- 生成失败状态。
- 加载状态。

## 19. 成功标准

第一版上线后，如果满足这些条件，就算 MVP 成功：

- 用户可以在 1 分钟内从长文生成阅读页。
- 生成结果明显比原始长文更容易阅读。
- 用户可以很快用 Google 登录。
- 每个生成结果都有稳定 ID。
- 登录后可以重新打开自己的生成记录。
- LLM 成本足够低，可以支撑免费试用。
